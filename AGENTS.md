# AGENTS.md — 광주새백성교회 예배 관리 프로그램

이 파일은 Claude Code, Cursor 등 AI 코딩 에이전트가 이 저장소에서 작업할 때 반드시
따라야 할 규칙이다. 무엇을 만드는지는 `PROJECT.md`를 참조하고, 이 파일은 어떻게
작업하는지를 다룬다.

## 0. 시작하기 전에

1. `PROJECT.md`를 전체 읽는다.
2. `PROJECT.md` §10 "미해결 사항"을 확인한다. 거기 체크되지 않은 항목과 관련된 작업을
   시작해야 하는 상황이면, **추측으로 코드를 짜지 말고 사용자에게 먼저 질문**한다.
   특히 다음은 확실해질 때까지 절대 임의로 정하지 않는다:
   - 주일찬양예배 구역 순서
   - 찬송가/교독문 파일명 규칙
   - 메일머지 필드명 (`F_`, `R_A_`, `R_B_`, `R_C_` 접두사 체계)
3. 기존 코드(`hwp_engine.py`, `ppt_engine.py` 등)가 저장소나 사용자로부터 제공되면
   그것을 신뢰하고 그대로 재사용한다. 같은 기능을 처음부터 새로 작성하지 않는다.

## 1. 작업 순서 — 한 번에 전체를 짜지 않는다

`PROJECT.md` §9의 로드맵 순서를 따른다. 이번 세션에서 몇 단계까지 할지 사용자가
명시하지 않았다면, 먼저 "어디까지 할까요?"라고 묻는다.

1. 기존 자산 이식 (한글 생성 + 주일낮예배 PPT 생성)
2. 찬양예배 PPT 확장
3. 성경본문 자동화
4. 실행 제어 기능 (찬송가/교독문/OBS+Snaz)
5. 설정/이력 화면 및 마감

각 단계가 끝나면:

- 무엇을 만들었는지 간단히 요약한다.
- win32com을 쓰는 코드(한글/PPT 생성)는 **이 환경(샌드박스)에서 직접 실행 검증이
  불가능하다**는 점을 분명히 한다. 문법 오류·임포트 오류 정도만 정적으로 확인하고,
  "실제 Windows PC에서 테스트가 필요합니다"라고 안내한다.
- subprocess/os.startfile을 쓰는 실행 제어 코드도 마찬가지로 실제 PC에서만
  검증 가능함을 안내한다.

## 2. 구역-마커 설정은 하드코딩 금지

`PROJECT.md` §3.3에 정의된 대로, 낮예배/찬양예배의 구역 구성(마커, 타입, 필드명)은
반드시 `app/section_config/*.json`으로 분리한다. 새로운 예배 종류가 추가되거나 구역
순서가 바뀌어도 Python 코드를 수정하지 않고 JSON만 바꾸면 되도록 만든다.

## 3. 저작권 — 성경 본문 처리 시 반드시 준수

`PROJECT.md` §3.4를 반드시 지킨다.

- 개역개정 본문을 DB에 통째로 저장하거나 캐싱하는 코드를 작성하지 않는다.
- 구절을 조회하면 그 자리에서 슬라이드 텍스트로만 변환하고 영구 저장하지 않는다.
- 교회가 자체 보유한 본문 데이터가 있는지 먼저 확인하고, 있다면 그것을 로컬 조회로
  쓰는 방식을 우선 제안한다 (외부 크롤링보다 안전하고 빠름).

## 4. 코드 스타일

- 백엔드: Python, FastAPI. 타입 힌트를 명시한다 (Pydantic 모델 적극 활용).
- 경로/시크릿은 `.env`로 관리하고 코드에 하드코딩하지 않는다.
- COM 객체(win32com)는 작업 후 반드시 해제하고 `Quit()`/`Close()`를 호출한다.
  예외 발생 시에도 `finally` 블록에서 정리되도록 작성한다 (PowerPoint/한글 프로세스가
  좀비로 남는 것을 방지).
- 프런트엔드: Bootstrap 5 + Vanilla JS 기본. SPA 프레임워크(React/Vue)는 도입하지
  않는다. 상태관리가 꼭 필요하면 Alpine.js 정도까지만 검토하고, 도입 전 사용자에게
  먼저 묻는다.
- 디자인 토큰은 `PROJECT.md` §5의 색상표/타이포 값을 그대로 CSS 변수로 박아 넣는다
  (임의로 다른 색상을 쓰지 않는다).

## 5. 에러 처리 원칙

- 파일/폴더 경로가 잘못된 경우 사용자가 바로 알 수 있게 구체적인 메시지를 준다
  (예: "찬송가 PPT 폴더에 '찬005_'로 시작하는 파일이 없습니다" — "오류 발생"처럼
  애매하게 표시하지 않는다).
- 찬송가/교독문 파일 매칭이 2개 이상이면 자동으로 아무거나 고르지 않고 사용자에게
  선택지를 보여준다.
- OBS/Snaz 등 실행형 기능은 실패해도 서버 전체가 죽지 않도록 격리해서 처리한다.

## 6. 하지 말아야 할 것

- 메일머지 필드명이나 파일명 규칙을 추측해서 임의로 만들어내지 않는다 — 모르면 질문한다.
- 성경 본문을 DB화하거나 정적 파일로 통째로 저장하지 않는다.
- PHP 프런트엔드 코드를 재사용하거나 참고하지 않는다 (디자인까지 포함해 신규로 만든다).
- win32com 관련 코드를 "샌드박스에서 실행해서 검증했다"는 식으로 단정하지 않는다 —
  실행 환경의 한계를 정직하게 알린다.
