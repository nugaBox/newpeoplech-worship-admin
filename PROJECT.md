# 광주새백성교회(newpeoplech) 예배 관리 프로그램 — 프로젝트 사양서

> 버전 0.1 (Draft) · 2026-06-21
> 이 문서는 AI 코딩 에이전트(Claude Code, Cursor 등)가 구현 작업의 기준으로 삼는 사양서다.
> 작업 방식/규칙은 `AGENTS.md`를 따르고, 이 문서는 "무엇을 만드는가"에 집중한다.

---

## 0. 한눈에 보기

매주 반복되는 예배 준비 작업(주보 작성, PPT 조립, 찬송가/교독문 파일 실행, 송출 프로그램 실행)을
하나의 로컬 웹 대시보드로 자동화한다.

- **형태**: 네이티브 윈도우 앱이 아니라 **같은 PC 안에서 도는 로컬 웹서비스**.
  - 서버: 예배 준비실 PC에서 FastAPI가 상시 백그라운드 실행 (Windows 작업 스케줄러 또는 NSSM 서비스 등록)
  - 클라이언트: 같은 PC의 크롬 브라우저에서 `http://localhost:8000` 접속
  - 문서 생성 기능(한글/PPT)은 win32com으로 해당 프로그램을 직접 제어
  - 실행 제어 기능(찬송가 켜기, OBS 켜기 등)은 같은 PC 위에서 subprocess로 로컬 프로그램 실행
  - **"같은 PC에서만 쓴다"가 전제** — 다른 PC/휴대폰에서의 원격 제어는 이번 범위 밖
- **이유**: 문서 생성과 로컬 프로그램 실행은 본질적으로 같은 종류의 작업(서버가 OS 기능을 호출)이므로
  별도의 네이티브 런처를 만들 필요가 없다. 웹 스택을 쓰면 디자인 자산(Bootstrap, Font Awesome,
  Pretendard)도 그대로 활용 가능하다.

### 기존 자산과의 관계 (중요)

이 프로젝트는 처음부터 새로 만드는 것이 아니라, 이미 동작 검증이 끝난 **기존 주보 자동생성
시스템**(FastAPI + win32com, 한글 메일머지 필드 치환 + 주일낮예배 PPT 구역 마커 삽입)을
**그대로 재사용·확장**하는 것이다.

| 항목 | 기존 시스템 | 이번 프로젝트 |
|---|---|---|
| 백엔드 | FastAPI (Python) | 동일 — 그대로 확장 |
| 한글 생성 | win32com `AllReplace`/`PutFieldText` | 동일 엔진 재사용 |
| PPT 생성(낮예배) | win32com `InsertFromFile`, 구역 마커 방식 | 동일 엔진 재사용, 구역 정의만 설정파일로 분리 |
| PPT 생성(찬양예배) | 없음 | 신규 — 동일 엔진을 다른 구역 셋으로 재사용 |
| 프런트엔드 | 기존 PHP 대시보드에 페이지 추가 | 신규 — Bootstrap + Pretendard 단일 웹앱으로 재구성 (PHP 폐기) |
| 실행 제어(찬송가/OBS 등) | 없음 | 신규 — subprocess 기반 로컬 실행 모듈 |

> **AI 에이전트에게**: 기존 `hwp_engine.py`, `ppt_engine.py` 코드가 제공되면 그것은
> 이미 운영 환경에서 동작 검증된 코드이므로, 새로 작성하지 말고 그대로 가져와
> `app/engines/`에 배치하고 거기서부터 확장할 것.

## 1. 배경 및 목표

### 1.1 현재 상황 (자동화 전)

매주 반복되는 수동 작업:

1. 주보 한글 파일을 열어 날짜·찬송가·설교 제목 등을 한 칸씩 직접 입력
2. 주일낮예배 PPT를 열어 해당 주차 찬송가/교독문/설교/광고 PPT를 일일이 복사해서 끼워 넣음
3. 주일찬양예배 PPT도 동일하게 반복
4. 예배 중 즉흥적으로 필요한 찬송가/교독문은 탐색기에서 파일을 찾아 더블클릭 실행
5. 예배 송출을 위해 OBS와 Snaz를 매번 수동으로 실행

→ 반복 작업, 오타/누락 실수, 긴 준비 시간이 문제.

### 1.2 목표

- 주차 예배 목차(찬송가 번호, 설교 제목, 본문 등)를 한 번만 입력하면 주보(한글)와
  PPT 2종(낮예배/찬양예배)이 자동 생성된다.
- 예배 진행 중에는 화면 버튼만으로 찬송가/교독문 슬라이드쇼를 즉시 실행한다.
- 예배 송출 준비(OBS+Snaz)를 버튼 하나로 끝낸다.
- 디자인은 무채색 기반의 절제된 톤(OpenAI 스타일 참고), Pretendard + Bootstrap + Font Awesome.

### 1.3 비목표 (이번 범위 제외)

- 다른 PC·휴대폰에서의 원격 제어 (실행 제어 기능은 같은 PC 전제; 구조는 추후 확장 가능하게 열어둠)
- 사용자 계정/권한 관리 (소규모 운영 PC 1대 기준, 로그인 없이 사용)
- 자동 자막/실시간 가사 송출 (OBS 내부 설정 영역 — 이 프로그램은 OBS "실행"까지만 담당)

## 2. 정보구조(IA) 및 화면 구성

### 2.1 메뉴 구조

| # | 메뉴 | 목적 |
|---|---|---|
| ① | 대시보드(홈) | 이번 주차 상태 요약, 빠른 실행 버튼 |
| ② | 주보/PPT 생성 | 주차 목차 입력 → 한글 주보 + PPT 2종 생성 |
| ③ | 예배 중 실행 | 찬송가/교독문 즉시 실행, OBS+Snaz 실행 |
| ④ | 설정 | 경로/디렉토리 등록, 템플릿 관리 |
| ⑤ | 이력 | 지난 주차 생성 파일 목록 (재다운로드/재생성) |

### 2.2 ① 대시보드(홈)

"오늘 무엇을 해야 하는가"에 집중.

- 이번 주 날짜 + "주보/PPT 생성 완료 여부" 상태 카드 (미완료=빨간 점, 완료=회색 체크)
- 자주 쓰는 실행 버튼 카드형 고정 노출: `[찬송가 켜기]` `[교독문 켜기]` `[OBS+Snaz 켜기]`
- 최근 생성 파일 3개 미리보기 (파일명, 생성 시각, 다운로드 아이콘)

### 2.3 ② 주보/PPT 생성 화면

좌측 입력 폼(아코디언 구조 — 필드 60여 개로 많음), 우측/하단에 "생성하기" 고정 버튼.

**입력 섹션:**

1. **기본 정보** — 날짜, 주차, 호수, 인용글, 광고 1~5
2. **주일낮예배** — 경배찬송, 교독문 번호, 찬송1, 기도자, 본문, 설교제목, 설교자,
   봉헌찬송, 영광찬송, 차주기도
3. **주일찬양예배** — 인도자, 찬송, 기도자, 본문, 설교제목, 차주기도
4. **첨부** — 설교 PPT, 광고 PPT 업로드 (드래그앤드롭)

> 찬송가 입력 칸은 1차 구현에서 단순 텍스트로 시작. 추후 "번호 입력 시 제목 자동표시"
> 자동완성으로 개선 가능 (찬송가 메타데이터 테이블 필요, 이번 범위 아님).

**생성 결과:**

- `[주보 생성]` `[낮예배 PPT 생성]` `[찬양예배 PPT 생성]` 개별 또는 "전체 생성" 일괄 실행
- 진행 중 단계별 상태 표시(대기/진행중/완료/실패) — win32com 작업은 수 초~수십 초
  걸릴 수 있으므로 스피너 + 단계 텍스트(예: "찬송가 슬라이드 삽입 중...")로 멈춘 게
  아님을 알 수 있게 함
- 완료 후 각 파일 다운로드 버튼 + "탐색기에서 폴더 열기" 버튼 (같은 PC이므로 가능)

### 2.4 ③ 예배 중 실행 화면

빠르게 조작해야 하므로 버튼은 크게, 입력은 최소화.

**찬송가 켜기:**

- 숫자 입력 필드(또는 키패드형 UI) + `[켜기]` 버튼
- 입력 즉시 디렉토리에서 매칭 후보를 리스트로 표시
  (예: "1" 입력 → "찬001_만복의근원하나님.ppsx" 매칭, 강조 표시)
- `[켜기]` 클릭 시 서버가 해당 PC에서 파일을 기본 프로그램(PowerPoint)으로 열고 쇼 모드 전환
- 최근 실행한 찬송가 5개를 버튼으로 바로 아래 노출 (재실행 편의)

**교독문 켜기:** 동일 패턴 — 번호 입력 후 매칭 파일 켜기

**OBS + Snaz 켜기:**

- 버튼 하나로 OBS.exe → Snaz 실행 파일 순서로 실행
- 이미 실행 중인 프로세스는 중복 실행하지 않고 "이미 실행 중" 안내만 (프로세스명으로 탐지)
- 실행 실패 시 구체적 에러 메시지 + 설정 화면 바로가기 링크

### 2.5 ④ 설정 화면

표 형태로 항목/현재값/변경 버튼 나열.

| 설정 항목 | 설명 | 예시 |
|---|---|---|
| 한글(HWP) 실행 경로 | Hancom Office 설치 경로 | `C:\Program Files...\Hwp.exe` |
| 주보 템플릿 경로 | 메일머지 필드 포함 hwpx 템플릿 | `...\주보_템플릿.hwpx` |
| 낮예배 PPT 템플릿 | 구역 마커 포함 pptx | `...\낮예배_템플릿.pptx` |
| 찬양예배 PPT 템플릿 | 구역 마커 포함 pptx | `...\찬양예배_템플릿.pptx` |
| 찬송가 PPT 폴더 | "찬001_만복의근원하나님.pptx" 형식 파일 모음 | `...\찬송가 PPT\` |
| 찬송가 슬라이드쇼 폴더 | ppsx 파일 모음 | `...\찬송가 PPSX\` |
| 교독문 PPT 폴더 | 교독문 번호별 PPT | `...\교독문 PPT\` |
| OBS 실행 경로 | obs64.exe 경로 | `C:\Program Files\obs-studio\bin\64bit\obs64.exe` |
| Snaz 실행 경로 | Snaz 실행 파일 경로 | `C:\...\Snaz.exe` |
| 성경 검색 API/방식 | 개역개정 본문 조회 출처 (§3.4 참고) | - |

각 경로 항목은 `[폴더 찾아보기]` 버튼으로 입력하고, 저장 시 실제 파일/폴더 존재 여부를
검증해 즉시 피드백(✓ 정상 / ✗ 경로 없음)을 표시한다.

### 2.6 ⑤ 이력 화면

- 주차별 생성 파일(주보 hwpx, 낮예배 pptx, 찬양예배 pptx) 테이블
- 재다운로드, 재생성(같은 입력값으로 다시 생성) 버튼
- 입력했던 목차 값도 함께 저장 → "지난주 불러오기"로 고정값(영광찬송, 차주기도 등) 반복입력 절감

## 3. 기능 상세 명세

### 3.1 주보(한글) 생성

기존 시스템과 동일한 방식 사용.

- 템플릿 hwpx에 `{{필드명}}` 플레이스홀더가 있으면 찾기/바꾸기(`AllReplace`)로 치환
- 누름틀(필드)로 만든 경우 `PutFieldText`로 치환 (더 안정적, 동일 텍스트 중복 치환 문제 없음)
  — 신규 템플릿 제작 시 누름틀 방식 권장
- 필드 매핑은 기존 정리된 약 60여 개 필드 체계(`F_`, `R_A_`, `R_B_`, `R_C_` 접두사) 유지
- 입력값이 비어있는 필드는 빈 문자열로 치환(에러 없이 진행), 생성 후 "빈 필드 목록"을
  경고로 표시해 누락 방지

### 3.2 주일낮예배 / 주일찬양예배 PPT 생성

PowerPoint COM(win32com)으로 템플릿을 열고, 구역 마커 슬라이드를 찾아 라이브러리 PPT의
슬라이드로 교체하는 기존 엔진을 그대로 재사용. 찬양예배는 구역 구성만 다르게 정의.

**동작 순서:**

1. 템플릿 pptx를 PowerPoint로 연다
2. 구역 마커 슬라이드(`[[경배찬송]]` 등)를 순서대로 찾는다
3. 마커에 대응하는 입력값(예: 경배찬송="201")으로 찬송가 PPT 폴더에서 파일명을 매칭한다
   — 파일명이 "찬001_만복의근원하나님.pptx"처럼 "찬+번호_제목" 형식이므로, 번호만으로
   찾으려면 폴더 내 파일명을 스캔해 `찬{0으로 채운 3자리}_`로 시작하는 파일을 탐색하는
   방식이 안전하다 (제목까지 입력값으로 정확히 맞추는 건 비현실적)
4. `InsertFromFile`로 매칭된 슬라이드를 마커 위치에 삽입한다
5. 마커 슬라이드를 삭제한다
6. 성경봉독 구역은 라이브러리 파일이 없으므로, 입력된 본문 텍스트로 슬라이드를 직접
   생성한다 (§3.4 참고)
7. 설교/광고 구역은 사용자가 업로드한 파일을 그대로 삽입한다
8. 나머지 텍스트 자리표시자(`{{날짜}}` 등)를 치환한다
9. 다른 이름으로 저장한다 (파일명 예: `주일낮예배_2026-06-21.pptx`)

**주일낮예배 구역 구성 (기존 정의 유지):**

경배찬송 / 교독문 / 찬송2 / 대표기도 / 성경봉독 / 설교 / 봉헌찬송 / 광고

**주일찬양예배 구역 구성 (확정 필요 — §6 미해결 사항 참고):**

기존 메일머지 필드 체계(`R_B_` 접두사: 인도설교, 인도자, 찬송, 기도자, 본문, 강해표시,
설교제목)를 기준으로 보면 대략 다음과 같은 구성이 예상된다. **실제 PPT 템플릿 제작 시
교회 측과 함께 최종 확정할 것**:

찬양 인도 / 찬송 / 대표기도 / 성경봉독 / 설교 / 광고

### 3.3 구역-마커 정의는 코드와 분리한다 (중요)

낮예배/찬양예배의 구역 목록, 그리고 각 구역이 "라이브러리에서 찾는 타입"인지
"업로드 파일을 쓰는 타입"인지 "텍스트로 직접 생성하는 타입"인지는 예배 순서가 바뀔 수
있으므로 **Python 코드에 하드코딩하지 말고 설정 파일(JSON)로 분리**한다.

```json
{
  "주일낮예배": [
    {"marker": "[[경배찬송]]", "type": "hymn_lib", "field": "경배찬송"},
    {"marker": "[[교독문]]",   "type": "responsive_lib", "field": "교독문번호"},
    {"marker": "[[찬송2]]",   "type": "hymn_lib", "field": "찬송1"},
    {"marker": "[[대표기도]]", "type": "upload",  "field": "대표기도파일"},
    {"marker": "[[성경봉독]]", "type": "scripture", "field": "본문"},
    {"marker": "[[설교]]",     "type": "upload",  "field": "설교파일"},
    {"marker": "[[봉헌찬송]]", "type": "hymn_lib", "field": "봉헌찬송"},
    {"marker": "[[광고]]",     "type": "upload",  "field": "광고파일"}
  ],
  "주일찬양예배": [
    "... 확정 후 동일한 형태로 정의 ..."
  ]
}
```

이렇게 하면 PPT 생성 엔진은 "타입별 처리 함수"만 가지고 있으면 되고, 구역 순서/추가/삭제는
설정 화면에서 편집 가능하도록 확장하기도 쉽다.

**타입별 처리:**

| type | 의미 | 처리 |
|---|---|---|
| `hymn_lib` | 찬송가 PPT 라이브러리에서 번호로 검색 | 찬송가 PPT 폴더에서 `찬{3자리}_` 접두 매칭 |
| `responsive_lib` | 교독문 PPT 라이브러리에서 번호로 검색 | 교독문 PPT 폴더에서 매칭 |
| `upload` | 사용자가 업로드한 파일 그대로 사용 | 업로드된 파일을 그대로 삽입 |
| `scripture` | 본문 텍스트로 슬라이드 직접 생성 | §3.4 로직 적용 |

### 3.4 성경본문(개역개정) 자동 검색 및 슬라이드 생성

**저작권 유의사항 (반드시 준수):** 개역개정 성경 본문은 저작권이 있는 번역본이므로,
별도 라이선스 계약 없이 외부 API/사이트를 크롤링해 통째로 가져다 쓰는 것은 저작권
문제가 발생할 수 있다.

- 매주 예배용으로 **그때그때 조회해서 그 주차 슬라이드에만 사용**(1회성 인용)하는 방식은
  일반적인 교회 활동 범위로 통용되나, 이를 **데이터베이스 형태로 통째로 복제·재배포하는
  것은 별개 문제이므로 피해야 한다**
- 권장 구조: (a) 사용자가 본문 구절 범위(예: "요한복음 3:16")만 입력 →
  (b) 서버가 신뢰할 수 있는 출처 한 곳에서 해당 구절만 조회 →
  (c) 결과를 슬라이드 텍스트로 바로 변환하고 **별도 영구 저장은 하지 않음**
- 만약 교회 자체에 이미 보유한 개역개정 본문 데이터(텍스트/엑셀 등)가 있다면, 외부
  검색 대신 **로컬 데이터로 조회**하는 것이 가장 안전하고 빠르다 — 이 경우 인터넷
  검색 자체가 불필요해짐 (구현 우선순위 1순위로 검토할 것)

**슬라이드 생성 로직 (출처와 무관하게 공통):**

- 구절 입력 → 절별로 줄바꿈된 슬라이드 자동 생성
- 한 슬라이드에 너무 많은 절이 들어가지 않도록 절 수 기준으로 슬라이드 분할
  (가독성 기준 — 정확한 절 수 임계값은 실제 템플릿 폰트 크기 확인 후 결정)

### 3.5 찬송가 슬라이드쇼 실행

- 입력 번호를 3자리로 패딩(예: `1` → `001`)
- 설정된 ppsx 폴더에서 `찬{번호}_`로 시작하는 파일 탐색
- 매칭 0개 → "파일을 찾을 수 없습니다" 안내
- 매칭 1개 → 즉시 실행
- 매칭 2개 이상 → 목록을 보여주고 사용자가 선택 (파일명 중복 방지는 별도 운영 규칙으로 안내)
- 실행은 `os.startfile()`로 ppsx의 기본 연결 프로그램(PowerPoint 쇼 모드)을 호출
  — ppsx 확장자 자체가 "열기 전용 쇼 모드"이므로 별도 매크로 없이 더블클릭과 동일 효과

### 3.6 교독문 PPT 켜기

- 찬송가와 동일 패턴, 대상 폴더와 파일명 규칙만 다름
- 교독문 PPT가 ppsx가 아닌 pptx일 수 있으므로, "쇼 모드로 바로 시작"하려면:
  - (A) PowerPoint COM으로 `SlideShowSettings.Run()` 호출, 또는
  - (B) 운영상 더 간단하게 교독문 PPT도 ppsx로 미리 내보내 두는 방식
  - **(B)를 권장** (구현 복잡도가 낮음 — 단, 교회 측에서 매번 ppsx로 변환해두는
    운영 부담이 있으므로 실제 운영자와 상의 후 최종 결정)

### 3.7 OBS + Snaz 실행

- `psutil`로 현재 실행 중인 프로세스 목록을 조회해 OBS/Snaz 실행파일명이 이미 떠 있는지 확인
- 떠 있지 않으면 `subprocess.Popen()`으로 실행, 떠 있으면 "이미 실행 중" 안내만
- OBS는 실행 후 켜지는 데 수 초 걸릴 수 있으므로, 실행 요청 자체는 비동기로 처리하고
  프런트에서는 "실행 요청을 보냈습니다"로 안내
  (실제 창이 떴는지까지는 1차 범위에서 확인하지 않음 — 필요시 추후 프로세스 폴링으로 보강)

## 4. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 백엔드 | Python · FastAPI | 기존 시스템과 동일, 코드 재사용 |

| 폰트 | Pretendard | CDN 또는 자체 호스팅 |
| 실행 방식 | uvicorn 상시 구동 | Windows 작업 스케줄러 또는 NSSM, 부팅 시 자동 시작 권장 |
| 데이터 저장 | SQLite 또는 JSON 파일 | 설정값/이력 저장 — 별도 DB 서버 불필요 |

> 프런트엔드는 React/Vue 같은 SPA 프레임워크 없이 Bootstrap + 약간의 JS로 충분하다.

> 다만 입력 폼 상태 관리가 번거로우면 가벼운 Alpine.js를 추가해도 무방 — 이 판단은
> 구현 단계에서 결정.

## 9. 단계별 구현 로드맵

전체 기능을 동등하게 본 사양서에 담았지만, 실제 구현은 **리스크가 낮고 기존 자산을
재사용하는 순서**로 진행한다.

1. **기존 자산 이식** — 기존 한글 생성, 주일낮예배 PPT 생성 엔진을 새 프로젝트 구조로
   옮기고 새 디자인의 프런트엔드 연결
2. **찬양예배 PPT 확장** — 주일찬양예배 구역 구성 확정 → `section_config` 추가 →
   동일 엔진으로 생성 검증
3. **성경본문 자동화** — 조회 출처 확정 → 구절 입력 → 슬라이드 자동 생성 로직 구현
4. **실행 제어 기능** — 찬송가/교독문 ppsx 실행 → OBS+Snaz 실행 → 프로세스 중복 탐지
5. **설정/이력 화면 및 마감** — 경로 설정 UI, 유효성 검증, 생성 이력, 디자인 다듬기
